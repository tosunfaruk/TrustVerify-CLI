## 1. Project Objective
The objective of this project is to develop a Python-based Command Line Interface (CLI) tool that enables a "Sender" to sign a file and a "Receiver" to verify its integrity and origin. [cite: 32] [cite_start]This is achieved by implementing hashing (SHA-256) for integrity and digital signatures (RSA) for authenticity.

## 2. Technical Implementation
* **Hashing (SHA-256):** The tool generates a unique SHA-256 hash for any given file (text, PDF, or image) to detect modifications. 
* **Manifest Generation:** A `metadata.json` file is created by scanning a directory, storing filenames alongside their respective hashes. 
* **Integrity Check:** A function compares the current state of files against the manifest to detect "Data Poisoning" or unauthorized tampering. 
* **Digital Signatures (RSA):** Using the `cryptography` library, a Public/Private key pair is generated for the user. 
* **Signing & Verification:** The tool encrypts the hash of `metadata.json` using the Sender's Private Key.  A second user can then use the Sender's Public Key to verify that the manifest is authentic and unaltered. 

## 3. Theoretical Analysis

### Q1: Why is hashing alone not enough to prove identity? 
**Answer:** While hashing ensures **Integrity** by detecting if a file has been changed, it does not provide **Authenticity**.  An attacker could modify both the file and the `metadata.json` file simultaneously.  Since the receiver only checks if the file matches the provided hash, they would have no way of knowing if the hash itself was generated by the legitimate sender or a malicious actor.  Hashing lacks a "link" to a specific identity. 

### Q2: How does the private/public key relationship ensure non-repudiation? 
**Answer:** Non-repudiation is ensured because a digital signature can only be created using the sender's **Private Key**, which is kept secret and accessible only to them. Since the signature can be verified by anyone using the corresponding **Public Key**, a successful verification proves that the signature must have been created by the owner of that specific private key. Therefore, the sender cannot later deny having signed or sent the document. 

## 4. Conclusion
"TrustVerify" successfully demonstrates the dual requirements of modern cybersecurity: ensuring that data remains unchanged (Integrity) and confirming the true identity of the source (Authenticity) through Digital Signatures.
