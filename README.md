## 1. Project Objective
[cite_start]The objective of this project is to develop a Python-based Command Line Interface (CLI) tool that enables a "Sender" to sign a file and a "Receiver" to verify its integrity and origin. [cite: 32] [cite_start]This is achieved by implementing hashing (SHA-256) for integrity and digital signatures (RSA) for authenticity. [cite: 33]

## 2. Technical Implementation
* [cite_start]**Hashing (SHA-256):** The tool generates a unique SHA-256 hash for any given file (text, PDF, or image) to detect modifications. [cite: 35]
* [cite_start]**Manifest Generation:** A `metadata.json` file is created by scanning a directory, storing filenames alongside their respective hashes. [cite: 36]
* [cite_start]**Integrity Check:** A function compares the current state of files against the manifest to detect "Data Poisoning" or unauthorized tampering. [cite: 37]
* [cite_start]**Digital Signatures (RSA):** Using the `cryptography` library, a Public/Private key pair is generated for the user. [cite: 38]
* [cite_start]**Signing & Verification:** The tool encrypts the hash of `metadata.json` using the Sender's Private Key. [cite: 39] [cite_start]A second user can then use the Sender's Public Key to verify that the manifest is authentic and unaltered. [cite: 40]

## 3. Theoretical Analysis

### [cite_start]Q1: Why is hashing alone not enough to prove identity? [cite: 41]
[cite_start]**Answer:** While hashing ensures **Integrity** by detecting if a file has been changed, it does not provide **Authenticity**. [cite: 42] [cite_start]An attacker could modify both the file and the `metadata.json` file simultaneously. [cite: 43] [cite_start]Since the receiver only checks if the file matches the provided hash, they would have no way of knowing if the hash itself was generated by the legitimate sender or a malicious actor. [cite: 44] [cite_start]Hashing lacks a "link" to a specific identity. [cite: 44]

### [cite_start]Q2: How does the private/public key relationship ensure non-repudiation? [cite: 45]
[cite_start]**Answer:** Non-repudiation is ensured because a digital signature can only be created using the sender's **Private Key**, which is kept secret and accessible only to them. [cite: 46] [cite_start]Since the signature can be verified by anyone using the corresponding **Public Key**, a successful verification proves that the signature must have been created by the owner of that specific private key. [cite: 47] [cite_start]Therefore, the sender cannot later deny having signed or sent the document. [cite: 48]

## 4. Conclusion
[cite_start]"TrustVerify" successfully demonstrates the dual requirements of modern cybersecurity: ensuring that data remains unchanged (Integrity) and confirming the true identity of the source (Authenticity) through Digital Signatures. [cite: 49]
